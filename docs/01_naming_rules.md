# 命名規則 / Naming Rules

## 1. 目的

このファイルは、Analysis Engine Library（分析エンジンライブラリ）で使用する名前の付け方を統一するためのルールを定義する。

命名規則を統一することで、分析項目、検査、ゲーム、診断文、索引、関係データを長期的に管理しやすくする。

## 2. 基本方針

このプロジェクトでは、日本語名を主、英語名を補助として扱う。

ユーザーや開発者が理解しやすいように、英語名を使う場合は必ず日本語名を併記する。

例：

- Working Memory（ワーキングメモリ）
- Sustained Attention（持続的注意）
- Executive Function（実行機能）

## 3. 正式名称

各項目には、必ず正式名称を1つ設定する。

正式名称は、原則として日本語で登録する。

例：

- 持続的注意
- ワーキングメモリ
- 抑制制御
- 認知の柔軟性
- 処理速度

## 4. 英語名

英語名は、心理学・医学・認知科学などで一般的に使われる表現を優先する。

英語が複数ある場合は、最も一般的な表現を primary_english_name として登録し、その他は alias に登録する。

例：

正式名称：持続的注意  
英語名：Sustained Attention  
別名：持続注意、注意維持、Attention Maintenance

## 5. 別名

同じ概念を指す可能性がある言葉は、削除せず alias として保存する。

例：

正式名称：ワーキングメモリ  
別名：

- 作業記憶
- Working Memory
- WM

正式名称：抑制制御  
別名：

- 反応抑制
- Inhibitory Control
- Response Inhibition

## 6. 似ているが別概念の扱い

似ている言葉でも、意味が異なる場合は別項目として登録する。

例：

- 処理速度
- 反応速度
- 判断速度

これらは近いが、同一ではないため別項目にする。

処理速度：情報を処理する速さ  
反応速度：刺激に対する反応の速さ  
判断速度：選択や意思決定までの速さ

## 7. 階層関係

上位概念と下位概念を分ける。

例：

注意  
├─ 持続的注意  
├─ 選択的注意  
├─ 分割的注意  
└─ 注意の切り替え

実行機能  
├─ 抑制制御  
├─ ワーキングメモリ  
├─ 認知の柔軟性  
├─ 計画性  
└─ 優先順位付け

## 8. 日本語表記のルール

原則として、自然な日本語を優先する。

ただし、心理学・医学・認知科学で一般的な用語は、そのまま使用する。

例：

使用する：

- ワーキングメモリ
- 実行機能
- 処理速度
- 抑制制御

避ける：

- 働く記憶
- 実行する能力
- 情報処理の速さ能力

## 9. 英語表記のルール

英語名は Title Case を基本とする。

例：

- Working Memory
- Sustained Attention
- Cognitive Flexibility
- Inhibitory Control

ファイル名やIDでは英語名を使わず、IDを使う。

## 10. ファイル名のルール

ファイル名にはIDを使う。

例：

- ITM-000001.md
- TST-000001.md
- GME-000001.md
- OUT-000001.md

項目名をファイル名に直接使わない。

理由：

- 日本語ファイル名による環境差を避ける
- 名前変更に強くする
- ID参照を安定させる

## 11. 非推奨の扱い

一度作成した項目名やIDは削除しない。

後から使用しない項目になった場合は、deprecated（非推奨）として扱う。

非推奨になっても、過去の参照のために残す。

## 12. 表示名と内部名

ユーザーに見せる名前と、内部で管理する名前は分ける。

例：

内部名：抑制制御  
表示名：ブレーキ力、待つ力、衝動コントロール

アプリの雰囲気に合わせて表示名を変えても、内部では同じIDを参照する。

## 13. 診断文での表現

診断文では、専門用語をそのまま出さない。

悪い例：

「あなたは抑制制御が低いです。」

良い例：

「今回のプレイでは、思いついた行動をすぐ試す場面が多く見られました。」

## 14. 命名時の優先順位

新しい項目名を決めるときは、以下の順で優先する。

1. 学術・医療・心理学で一般的な日本語名
2. 日本語として自然で意味が伝わる名前
3. ゲーム分析で使いやすい名前
4. アプリ上で面白く見せやすい名前

正式名称は1または2を優先し、3や4は表示名として扱う。

## 15. まとめ

このプロジェクトでは、名前は削除せず、IDを中心に管理する。

名前が増えても問題ないように、正式名称、英語名、別名、表示名を分けて扱う。

これにより、将来的にゲームやアプリが増えても、分析エンジン全体の一貫性を保つ。
